**Билет 14. Нормальная форма 6, Доменно-ключевая форма, денормализация**

### Шестая нормальная форма (6NF)
- **Требование**: отношение разлагается до атомарных фактов — каждая таблица содержит только один факт (первичный ключ + один атрибут).
- **Цель**: поддержка **временных данных** и устранение всех возможных зависимостей.
- **Пример**:  
  Вместо `Сотрудник(id, имя, должность)` → три таблицы:  
  ```sql
  CREATE TABLE сотрудники_имя (id INT PRIMARY KEY, имя TEXT);
  CREATE TABLE сотрудники_должность (id INT PRIMARY KEY, должность TEXT);
  ```
- **Нарушение 6NF**: любая таблица с >1 неключевым атрибутом.

### Доменно-ключевая нормальная форма (DKNF)
- **Требование**: все ограничения являются логическим следствием доменных и ключевых ограничений.
- **Практически недостижима**: требует полного описания бизнес-правил через домены и ключи.

### Денормализация
- **Определение**: сознательное нарушение нормальных форм для повышения производительности (уменьшение JOIN’ов).
- **Назначение**: оптимизация OLAP-запросов, кэширование вычисляемых данных.
- **Проблемы**:  
  - Риск аномалий обновления (несогласованность данных).  
  - Увеличение объёма хранилища.  
  - Сложность поддержки целостности (требуются триггеры или прикладная логика).

Денормализация — компромисс между чистотой модели и производительностью.

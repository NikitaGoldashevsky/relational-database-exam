**Билет 8. Графовая и Столбцовая NoSQL модели данных**

### Графовая модель
- **Структура**: узлы (сущности), рёбра (связи) и свойства на них.  
  Пример: `(:Person {name: "Alice"})-[:FRIEND]->(:Person {name: "Bob"})`.
- **Назначение**: анализ сложных связей — соцсети, рекомендации, мошенничество.
- **СУБД**: Neo4j, Amazon Neptune.
- **Достоинства**:  
  - Эффективные обходы связей (O(1) по сравнению с JOIN в SQL).  
  - Интуитивное представление отношений.
- **Недостатки**:  
  - Плохо масштабируется для аналитических запросов без связей.  
  - Отсутствие стандарта запросов (Cypher, Gremlin и др.).

### Столбцовая (колоночная) модель
- **Структура**: данные хранятся по столбцам, а не по строкам; группы столбцов объединяются в семейства.
- **Назначение**: OLAP, big data, агрегация по большим объёмам (например, логи, телеметрия).
- **СУБД**: Apache Cassandra, HBase, ScyllaDB.
- **Достоинства**:  
  - Высокая скорость агрегации и сжатия (однородные данные в столбце).  
  - Горизонтальное масштабирование и отказоустойчивость.
- **Недостатки**:  
  - Неэффективна для транзакций и частых обновлений.  
  - Сложность моделирования отношений.

Обе модели решают задачи, где классическая реляционная модель теряет производительность или гибкость.

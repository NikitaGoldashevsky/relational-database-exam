**Билет 7. Объектно-ориентированные и объектно-реляционные модели данных**

### Объектно-ориентированная модель (OODBMS)
- **Структура**: данные представлены как объекты с атрибутами, методами, наследованием и инкапсуляцией.
- **Расцвет**: конец 1980-х — 1990-е (например, ObjectStore, Versant).
- **Назначение**: хранение сложных структур (CAD, мультимедиа), где реляционная модель неудобна.
- **Достоинства**:  
  - Естественное соответствие ООП-коду.  
  - Поддержка сложных типов и поведения.
- **Недостатки**:  
  - Отсутствие стандарта запросов (в отличие от SQL).  
  - Сложность администрирования и масштабирования.  
  - Низкая совместимость с существующими реляционными системами.

### Объектно-реляционная модель (ORDBMS)
- **Структура**: расширение реляционной модели: пользовательские типы, наследование таблиц, методы.
- **Расцвет**: середина–конец 1990-х (PostgreSQL, Oracle 8i).
- **Назначение**: совмещение гибкости ООП и надёжности реляционного подхода.
- **Достоинства**:  
  - Сохраняется SQL и ACID.  
  - Поддержка сложных данных (геометрия, JSON, массивы).  
  - Расширяемость.
- **Недостатки**:  
  - Усложнение схемы и запросов.  
  - Не все СУБД реализуют OR-возможности одинаково.

**Пример в PostgreSQL**:
```sql
CREATE TYPE point AS (x INT, y INT);
CREATE TABLE figures (id SERIAL, center point);
```
